  ---
  MinCut, RuVector, and RVF — What They Do to the Agentic Workflow

  ---
  1. MinCut — Codebase Graph Partitioning

  MinCut is a real implementation using the Stoer-Wagner algorithm exposed as
  the MCP tool hooks_graph_mincut. It is not a swarm feature — it is a static 
  analysis pass that feeds agent routing decisions.

  What it does:

  It builds a directed dependency graph from your codebase's import/export
  statements, then finds the minimum set of edges to cut to split the graph into
   two natural groups. The result is:

  partition1: ["src/auth/", "src/users/"]      ← cohesive cluster A
  partition2: ["src/payments/", "src/orders/"] ← cohesive cluster B
  bridges: [{ from: "src/auth/index.ts", to: "src/payments/gateway.ts" }]
  articulationPoints: ["src/auth/index.ts"]    ← critical node — removing it
  splits the graph
  suggestion: "Natural boundary: auth/users vs payments/orders"

  What changes in your workflow when it is active:

  Without MinCut: Agent assignment is based only on file extension and task
    keywords
  With MinCut active: Agent assignment also uses which graph community the file
    belongs to — high-cohesion files get routed to specialized agents,
    bridge/articulation-point files get routed to architects
  ────────────────────────────────────────
  Without MinCut: Refactoring suggestions are generic
  With MinCut active: hooks_graph_mincut outputs a human-readable suggestion
    directly: "This is the natural boundary — split here"
  ────────────────────────────────────────
  Without MinCut: Task parallelization is arbitrary
  With MinCut active: The swarm coordinator splits tasks along the MinCut

  ────────────────────────────────────────
  Without MinCut: No structural risk awareness
  With MinCut active: Articulation points are flagged as high-risk — edits there

    affect both clusters

  Relation to the sublinear agents: MinCut is O(V³) and runs before the
  sublinear solvers. It reduces the graph size first, then the
  consensus-coordinator runs the Neumann series solver on each smaller
  partition's adjacency matrix. The pagerank-analyzer then scores each node
  within a partition for voting influence.

  2. RuVector (@ruvector/*) — What Each Package Does to the Workflow

  These are four WASM kernels, each targeting a different bottleneck:

  ---
  @ruvector/attention — Flash Attention Kernel

  The problem it solves: When you run a memory search over a large context,
  naive attention is O(N²) memory. With 10,000 stored patterns, that blows
  through RAM.

  What it actually does:

  - Processes attention in blocks of 32 vectors (fits L1 cache)
  - Uses sparse attention: only attends to the top 12% of keys (minimum 16,
  maximum 96) — ignoring irrelevant patterns entirely
  - 8x loop unrolling for dot products
  - Pre-allocated Float32Array buffers — no garbage collector pauses mid-query
  - Runs after HNSW returns top-K candidates — flash attention re-ranks them

  What changes:

  ┌─────────────────────────────────────┬───────────────────────────────────┐
  │               Before                │               After               │
  ├─────────────────────────────────────┼───────────────────────────────────┤
  │ Semantic re-ranking of 1000         │ ~47–140ms (2.49x–7.47x speedup)   │
  │ patterns: ~350ms                    │                                   │
  ├─────────────────────────────────────┼───────────────────────────────────┤
  │ O(N²) memory for full attention     │ O(N) memory via tiling            │
  ├─────────────────────────────────────┼───────────────────────────────────┤
  │ GC pauses during large queries      │ Zero GC pressure (pre-allocated   │
  │                                     │ buffers)                          │
  └─────────────────────────────────────┴───────────────────────────────────┘

  ---
  @ruvector/sona — Self-Optimizing Neural Architecture (LoRA + EWC++)

  The problem it solves: Every time you run the same class of task (e.g., "add
  TypeScript types to a JavaScript file"), the router has to reason from scratch
   which agent is best. SONA learns from outcomes and routes faster and more
  accurately over time.

  What it actually does:

  - On hooks_trajectory_end (task completes), extracts keywords from the task
  description
  - Updates a confidence score for the keyword → agent mapping: +0.1 on success,
   -0.15 on failure
  - Applies time decay: -0.01 per day — stale patterns lose authority
  - Persists to .swarm/sona-patterns.json
  - EWC++ (Elastic Weight Consolidation++): prevents catastrophic forgetting —
  when new patterns are learned, previously high-confidence patterns are not
  overwritten
  - LoRA (Low-Rank Adaptation): fine-tunes the routing weights with minimal
  compute

  What changes:

  ┌─────────────────────┬───────────────────────────────────────────────────┐
  │      Session 1      │           After 20+ sessions with SONA            │
  ├─────────────────────┼───────────────────────────────────────────────────┤
  │ Every task routes   │ High-confidence learned routes skip the heuristic │
  │ by keyword          │  and resolve in ~0.045ms                          │
  │ heuristic           │                                                   │
  ├─────────────────────┼───────────────────────────────────────────────────┤
  │ All "implement      │ Learned: "implement auth" in this codebase →      │
  │ auth" tasks → coder │ security-architect (because security-architect    │
  │                     │ succeeded more often)                             │
  ├─────────────────────┼───────────────────────────────────────────────────┤
  │ No learning from    │ Failed routes reduce confidence; alternatives are │
  │ failures            │  promoted                                         │
  ├─────────────────────┼───────────────────────────────────────────────────┤
  │ New agent session   │ SONA patterns persist across sessions — the       │
  │ starts cold         │ system remembers what worked                      │
  └─────────────────────┴───────────────────────────────────────────────────┘

  ---
  @ruvector/router — Q-Learning Semantic Router

  The problem it solves: Which of the 60+ agents should handle this task? The
  router combines four signals rather than guessing.

  The four signals it blends:

  1. SONA pattern match — learned keyword → agent mappings (confidence-weighted)
  2. Q-learning table — state (task_type + file_context) → best action (agent),
  updated by reward on completion
  3. Coverage signal — if test coverage for the affected file is below
  threshold, tester agent gets priority
  4. MinCut graph signal — which community the file belongs to, and whether it
  is an articulation point

  What changes:

  Without router:  "implement OAuth" → coder  (always)
  With router:     "implement OAuth" in src/auth/ (articulation point, coverage
  34%)
                   → security-architect (SONA: 0.87 confidence) + tester spawned
   in parallel
                      because: coverage signal overrides, graph signal flags it
  as boundary node

  The returned RouteResult includes confidence, reason, and alternates — Claude
  Code can see why a routing decision was made.

  ---
  @ruvector/learning-wasm — Neural Training Kernel

  Used by the background ultralearn worker and neural train command. Runs the
  JUDGE → DISTILL → CONSOLIDATE pipeline:
  - JUDGE: was this trajectory a success? (binary verdict, stored with
  embedding)
  - DISTILL: extract a reusable pattern via LoRA weight extraction
  - CONSOLIDATE: merge into long-term memory without overwriting other patterns
  (EWC++)

  This runs in the background — it does not block agent execution.

  3. RVF (RuVector Format) — Two Distinct Things

  RVF is used for two completely different purposes under the same name.

  ---
  RVF as a Database Format (Chat UI / ruvocal)

  A zero-dependency JSON document store that replaces MongoDB for the local chat
   UI (ruvocal). The entire database is a single JSON file:

  {
    "rvf_version": "2.0",
    "format": "rvf-database",
    "collections": {
      "conversations": { "id1": { ... }, "id2": { ... } },
      "messages": { ... }
    },
    "metadata": { "created_at": "...", "doc_count": 1847 }
  }

  - In-memory Map<id, Record> as primary structure → O(1) lookup
  - Writes debounced 500ms — no write thrash
  - Implements the full MongoDB Collection interface (findOne, find, insertOne,
  updateOne, aggregate, bulkWrite, and 15 MongoDB operators like $set, $push,
  $or, $regex, etc.)
  - All 56 existing HuggingFace Chat UI files work without modification

  What changes: Cold start drops from ~3.5s (MongoDB driver) to ~1.2s. No Docker
   dependency for persistence. The Docker compose mongodb service becomes
  optional.

  ---
  RVF as a Package/Deployment Segment Format

  The rvf.manifest.json describes ruflo's own deployment artifact as a segmented
   container:

  ┌─────────────┬───────────────────────────────┬───────────────────────────┐
